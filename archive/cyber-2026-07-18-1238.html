Top Story

WordPress patches wp2shell — a pre-auth RCE in core that needs no login, no plugins and no special configuration

WordPress released 6.9.5 and 7.0.2 on July 17, closing a pre-authentication remote code execution flaw in core. The Hacker News summarises the problem bluntly: an anonymous HTTP request can run code on a WordPress site, and because the bug is in core, "a bare install with zero plugins is exploitable." WordPress's own release post describes the finding as "a REST API batch-route confusion and SQL injection issue leading to Remote Code Execution."

The flaw was found by Adam Kues at Assetnote, the attack-surface-management arm of Searchlight Cyber, and reported through WordPress's HackerOne program. Searchlight's write-up, published under the name wp2shell, says the attack has "no preconditions and can be exploited by an anonymous user." The firm is withholding technical details for now and has published a checker at wp2shell.com instead.

Two release branches are affected: 6.9.0 through 6.9.4 (fixed in 6.9.5) and 7.0.0 through 7.0.1 (fixed in 7.0.2). Versions 6.8.5 and below are not affected, and 7.1 beta2 carries the same fix. A separate 6.8.6 release addresses a second SQL injection bug reported by a different team. WordPress has enabled forced updates through its auto-update system, but as THN notes, it has not said whether that push reaches sites that turned auto-updates off — so verify what you are actually running.

Two details make this harder to manage than a normal critical bug. First, neither advisory carries a CVE ID or a CVSS score, and no CVE record had appeared by July 18 — meaning CVE-keyed scanners will not flag it and CISA cannot add it to KEV until an identifier exists. Track it by version number instead. Second, the patch is the map: WordPress core is open source and both 7.0.1 and 7.0.2 sit in the public release archive, so anyone can diff the fix. When Searchlight turned Drupal's public core SQL-injection fix into a working teardown in May, it took a single day.

Searchlight estimates over 500 million websites run WordPress — but that is the total install base, not the vulnerable population. The flawed code only exists from 6.9 onward, which shipped December 2, 2025, so every affected site is running a release less than eight months old. Neither advisory says how many sites that covers. No exploitation attempt had been reported as of July 18.

Patch Priority

⚠ Most urgent action today — federal deadline falls tomorrow

Microsoft SharePoint Server — CVE-2026-58644 (CVSS 9.8). A critical deserialization-of-untrusted-data flaw allowing remote code execution. Microsoft revised its bulletin to confirm the flaw was exploited in the wild as a zero-day before patches shipped in the July 14 Patch Tuesday release. CISA added it to the KEV catalog on July 16 and requires Federal Civilian Executive Branch agencies to apply fixes by July 19, 2026.

Affected: SharePoint Server Subscription Edition, SharePoint Server 2019, and SharePoint Enterprise Server 2016. Microsoft warns the flaw is remotely exploitable over the internet with low attack complexity, because an attacker needs no significant prior knowledge of the system and can achieve repeatable success with the payload.

The same July 19 deadline applies to two actively exploited Fortinet FortiSandbox OS command injection flaws, CVE-2026-25089 and CVE-2026-39808, also added on July 16.

Then update WordPress to 6.9.5 or 7.0.2. If you cannot patch today, Searchlight's mitigations all come down to keeping anonymous callers off the batch endpoint — and it is explicit that a WAF rule must block both /wp-json/batch/v1 and rest_route=/batch/v1, because covering only the path leaves the query-string route open.

Threat Actor Spotlight

Mass exploitationWordPress

WP-SHELLSTORM

The crew that illustrates why this week's WordPress bug matters. Before its server leaked in June, WP-SHELLSTORM got into more than 17,000 sites by its own count — through a single caching-plugin flaw that was already public, already patched, and only worked on a non-default setting. The wp2shell flaw is in core, needs no plugin, and works against a default install. As The Hacker News puts it, mass exploitation of WordPress is an industry now.

Breaches & Incidents

Crypto theftCoinspect

"Ill Bloom" wallet flaw drains over $5 million

Security firm Coinspect disclosed a flaw in how some wallet software generated recovery phrases. Where weak randomness was used, an attacker can derive the phrase and take everything it controls. Coinspect confirmed one coordinated sweep on May 27 that drained about $3.1 million from 431 wallets, and told The Hacker News a further $2.1 million in USDT was stolen from an exposed wallet afterward. Risk sits with older or lesser-known mobile and browser-extension wallets, some dating to 2018; hardware-device wallets are not affected.

SharePointCISA advisory

CISA urges SharePoint hardening after new exploitations

CISA warned of active exploitation of multiple SharePoint Server flaws — CVE-2026-32201, CVE-2026-45659, CVE-2026-56164 and CVE-2026-58644 — affecting all supported on-premises versions. Attacks involve remote code execution plus post-exploitation activity such as stealing IIS machine keys and deserialization techniques to gain persistence and deploy malware.

CISA KEV & Federal Deadlines

  • CVE-2026-58644 — Microsoft SharePoint Server deserialization → RCE. Added July 16; FCEB agencies must remediate by July 19, 2026.—
  • CVE-2026-25089 and CVE-2026-39808 — Fortinet FortiSandbox OS command injection. Added July 16; agencies have until July 19, 2026 to update to the latest supported versions.—
  • July 14 batch — CISA added four vulnerabilities including CVE-2026-15409 (SonicWall SMA1000 SSRF), CVE-2026-15410 (SonicWall SMA1000 code injection) and CVE-2026-56155 (Microsoft AD FS access control).
  • July 1 — CVE-2026-45659 (Microsoft SharePoint Server deserialization) added on evidence of active exploitation; it remains one of the four SharePoint flaws in CISA's July 14 hardening warning.
  • Remediation deadlines are assigned per-CVE under BOD 26-04, which requires agencies to prioritise rapid remediation of high-risk KEV entries — publish the CISA-stated due date rather than assuming a fixed window.

CISA's Recommended Hardening

  • Apply the latest Microsoft patches, verify they installed successfully, and shorten patching cycles where possible.
  • Verify that AMSI integration is enabled for each SharePoint web application.
  • Scan for and remove intrusion artifacts — including machine-key harvesting tools — before rotating IIS machine keys, to avoid the new keys being stolen too.
  • Establish tailored logging to detect and monitor exploitation activity.
  • Avoid exposing SharePoint Servers directly to the internet unless necessary; block external access to SharePoint Central Administration and restrict farm and database communications to required systems.
